GitHub flow est un workflow basé sur une branche qui prend en charge les équipes et les projets dans lesquels des déploiements ou des merges sont effectués régulièrement.

Créer une branche

Lorsque vous travaillez sur un projet, vous allez avoir à tout moment un tas de fonctionnalités ou d'idées différentes - dont certaines sont prêtes à être utilisées, et d'autres qui ne le sont pas. La branche existe pour vous aider à gérer ce flux de travail.

Lorsque vous créez une branche dans votre projet, vous créez un environnement dans lequel vous pouvez essayer de nouvelles idées. Les modifications que vous effectuez sur une branche n'affectent pas la branche principale, vous êtes donc libre d'expérimenter et de valider les modifications, sachant que votre branche ne sera pas fusionnée tant qu'elle ne sera pas prête à être examinée par une personne avec laquelle vous collaborez

Astuces

Branching est un concept central dans Git, et l'ensemble du flux GitHub est basé sur lui. Il n'y a qu'une seule règle: tout ce qui se trouve dans la branche master est toujours déployable.

Pour cette raison, il est extrêmement important que votre nouvelle branche dev ou feature soit créée hors du master lorsque vous travaillez sur une fonctionnalité ou un correctif. Le nom de votre branche doit être descriptif (par exemple, Home-section,removing-slide), afin que les autres puissent voir ce sur quoi on travaille.

Commits

Une fois la branche feature crée , on va procéder à faire nos changements chaque modification est suivi d’un commit qui sont automatiquement ajouté à la branche feature , l’utilité est qu’on garde un système de tracking de l’avancement du projet

la particularité des commits, c’est la création d’un historique transparent que les autres collaborateurs peuvent le suivre pour comprendre le déroulement du projet , en effet chaque commit est considéré comme une portion séparé des changements, grâce à cette particularité on peut faire des « roll back » ou un appel des commits avec la commande git reset ou git revert

Astuces

Les messages des commits sont un premier filtre de la reconnaissance du projet , il doit être clair et pertinent

Pull Request

Le pull Request lance la discussion sur vos changements, tous les collaborateurs peuvent voir les changements que tu as effectué

On peut ouvrir le pull Request n’importe quelle moment dans le processus du développement, c’est un espace de partage d’idée, des Screenshots et spécialement si on a besoin d’un conseil d’un reviewer ou d’un collègue du groupe, en utilisant la mention @mention

Astuces

Les pull request sont utiles pour contribuer à des projets open source et pour gérer les modifications apportées aux Repositories partagés. Si vous utilisez un Fork, Pull Requests fournissent un moyen d'informer le responsable du projet de prendre en considération les modifications que vous proposez. Si vous utilisez un Repository partagé, les pull requests aident à démarrer la révision du code et la conversation sur les modifications proposées avant qu'elles ne soient fusionnées dans la branche master.

Reviewing

Une fois un pull request est ouvert, la personne ou l'équipe qui examine vos modifications peut avoir des questions ou des commentaires sur plusieurs sujets. Les pull requests sont conçues pour encourager ce type de conversation.

Si quelqu'un commente que vous avez oublié de faire quelque chose ou s'il y a un bug dans le code, vous pouvez le corriger dans votre branche local et faire push aux modification.

Astuces

Les commentaires sur les pull requests sont écrits sous format ‘Markdown’, vous pouvez donc ajouter des images et des emoji, utiliser des blocs de texte pré-formatés et d'autres formats légers.

Déploiement

Avec GitHub, vous pouvez déployer à partir de la branche feature pour les tests finaux en production avant de fusionner vers master

Une fois le pull request a été examinée , vous pouvez déployer vos modifications pour les vérifier en production. Si votre branche pose problème, vous pouvez la restaurer en déployant master existant en production.

Astuces

Un article optionnel descriptif sur le déploiement : GitHub Deploy

Merge

Maintenant que vos modifications ont été vérifiées, il est temps de fusionner votre code dans la branche master.

Une fois fusionnées, les pull request conservent un enregistrement des modifications historiques de votre code.

Astuces

En ajoutant certains mots clés dans le texte de votre Pull Request, vous pouvez associer des problèmes au code ou des issues. Lorsque votre pull request est fusionnée, les problèmes liés sont également fermés.
